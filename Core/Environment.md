# C++開発環境
C++で書かれたソースコードを実行ファイルに変換するまでのプロセスを整理しながら、2026年のC++開発環境について説明します。


## 登場人物

| ツール | カテゴリ | 役割 |
|--------|----------|------|
| C++ | 言語 | ソースコード |
| Clang | コンパイラ | フロントエンド処理・リンカ呼び出し |
| LLVM | コンパイラ基盤 | 最適化・機械語生成 |
| CMake | ビルドシステム | ビルド定義の記述 |
| Ninja | ビルドランナー | CMakeのルールを実行 |
| Clang-Format | 静的解析 | コードの整形 |
| Clang-Tidy | 静的解析 | バグパターン・ベストプラクティスの指摘 |
| CppCheck | 静的解析 | Clangと独立した静的解析エンジン |
| GoogleTest | テストフレームワーク | 単体テストの記述・実行 |




## コードを書く
C++のソースコードを.cppや.hppファイルに記述する。  
> [!NOTE]
> .cppファイルは.cxxや.ccなどの拡張子が使われることもある
> .hppファイルは.hや.hxx、.hhなどの拡張子が使われることもある

.hppファイルは宣言コード、.cppファイルには定義コードを含めるのが一般的である。  
このときに利用するツールとして、Clang-FormatやClang-Tidy、CppCheckなどの静的解析ツールがある。

Clang-Formatはソースコード整形ツールであり、プロジェクト全体でコードスタイルを統一するために使用する。
.clang-formatファイルにスタイルルールを定義しておくことで、ルールに従ってコードを自動的に整形できる。

Clang-Tidyはリンターツールであり、コードの品質を向上するために使用する。
スタイルガイドの違反やモダンC++のベストプラクティスに反するコードなどを指摘してくれる。
.clang-tidyファイルにチェックルールを定義しておくことで、ルールに従ってコードを自動的に解析できる。

CppCheckは静的コード解析ツールであり、バグが発生しやすいパターンを検出するために使用する。
潜在的なバグやパフォーマンスの問題、脆弱性の有無などを指摘してくれる。

これらのツールはいずれもエディタ上での実行やCI組み込みが可能であり、併用が推奨される。
役割としては、Clang-Formatはコードの整形、Clang-Tidyはコード品質の改善、CppCheckはバグの検出を担う。


## テストする
GoogleTestはC++のテストフレームワークであり、単体テストのコードを記述・実行するために使用する。
テストコードは.cppファイルに記述し、GoogleTestのマクロを使用してテストケースを定義する。
テストケースは、クラスや関数の動作を検証するためのコードであり、期待される出力や例外の有無などをチェックする。

ここで、C++のソースコードをテストするには、テスト対象となるコードをコンパイルする必要がある。
そこで、ソースコードとテストコードの両方をコンパイルしてオブジェクトファイルを生成し、それらをリンクしてテスト実行ファイルを生成する。
この実行ファイルを実行することで、テストケースが実行され、テスト結果を得ることができる。
このとき、本番コードとテストコードではエントリポイント（main関数）が異なるため、それぞれ別の実行ファイルになる。
テストコードと本番コードを分けて管理することで、本番と同様のコンパイル環境を使いながら、それぞれ異なるコンパイルオプションを指定する必要がある。


## コンパイルする
コンパイルとは、ソースコードを機械語（オブジェクトファイル）に変換する処理である。
おおきく分けて、ソースコード解析とIR生成、IR最適化、コード生成の3段階に分けられる。


### ソースコード解析とIR生成
記述したソースコードをコンパイルして実行ファイルに変換するためには、まずソースコードを解析して中間表現（Intermediate Representation、IR）を生成する必要がある。
この処理を行うツールがClangである。
また、このような処理をコンパイラのフロントエンドとよぶ。


#### 解析
まず、ソースコードを解析して、字句解析、構文解析、意味解析を行う。

字句解析とは、ソースコードをトークンに分割する処理である。
トークンとは、キーワードや識別子、リテラルなどの言語的に意味のある最小単位を指す。
例えば、`int x = 42;`というコードは、`int`、`x`、`=`、`42`、`;`という5つのトークンに分割される。

構文解析とは、トークン列を構文木に変換する処理である。
構文木とは、ソースコードの構造を表す木構造であり、ノードが文や式などの構成要素を表す。
このとき、構文エラー、つまり、文法や構文規則に反するコードがある場合はエラーが発生する。
例えば、`int x = 42`というコードは、セミコロンがないため、構文エラーとなる。

意味解析とは、構文木を解析して、型チェックや変数のスコープ解析などの処理を行う。
例えば、`int x = "hello";`というコードは、構文上は形式的に問題ないが、プログラムの意味としては、整数型の変数に文字列リテラルを代入しようとしているためエラーとなる。
このような型検査やスコープ解析、名前解決などを行うためにシンボルテーブルが使用される。
シンボルテーブルとは、変数や関数などの識別子とその型やスコープなどの情報を管理するデータ構造である。
例えば、`int x = 42;`というコードがある場合、シンボルテーブルには、識別子`x`とその型`int`、スコープ情報などが登録される。


#### IR生成
意味解析が完了した後、構文木を中間表現（Intermediate Representation, IR）に変換する処理が行われる。
人間が記述しやすい高級言語から機械語に近い低級言語への変換を一気に行うには、抽象度のギャップが大きすぎるため、極端な意味変化が発生したり、環境ごとの違いを吸収できず最適化が難しくなったりする。
そこで、直接アセンブリに変換するのではなく、環境に依存しない中間表現（IR）を生成する。

代表的な中間表現として、LLVM IRがある。
例えば、`int x = 42;`というコードは、LLVM IRでは、以下のようなコードに変換される。
```llvm
%x = alloca i32, align 4
store i32 42, ptr %x, align 4
```
対して、アセンブリ言語では、以下のようなコードに変換される。
```assembly
x:
    .long   42
```


### 最適化
生成されたIRは、元のソースコードの意味を保持しながら、比較的アセンブリに変換しやすく、かつ、環境に依存しない形で表現されている。
このIRを最適化することで、より効率的なコードを生成することができる。
この処理を行うツールがLLVMであり、LLVMで利用される中間表現がLLVM IRである。
また、このようなIR最適化処理をコンパイラのミドルエンドとよぶ。

LLVMは、IRを最適化するための様々な最適化パスを提供している。
例えば、関数インライン展開やループアンローリングなどがある。
関数インライン展開とは、関数呼び出しを関数の本体に置き換える最適化である。
ループアンローリングとは、ループの繰り返し回数を減らすために、ループの本体を複数回展開する最適化である。


### コード生成
最適化されたIRから、ターゲット環境に応じた機械語を生成する処理を行う。
この処理もLLVMが担当しており、コンパイラのバックエンドとよぶ。

バックエンドでは、IRをターゲットCPUの命令セット（x86-64、ARM64など）に対応した機械語に変換し、オブジェクトファイルとして出力する。
オブジェクトファイルとは、機械語とメタデータが含まれたファイルであり、関数や変数への参照がまだ未解決の状態で保持されている。
オブジェクトファイルのフォーマットはOSごとに異なり、WindowsではCOFF（.obj）、LinuxではELF（.o）、macOSではMach-O（.o）が使用される。


## ビルドする
ビルドとは、コンパイルを含む、ソースコードから実行ファイルを生成するまでの全工程を指す。
具体的には、コンパイル（ソースコード → オブジェクトファイル）とリンキング（オブジェクトファイル → 実行ファイル）を組み合わせた処理である。


### リンキング
生成されたオブジェクトファイルとライブラリを結合し、すべての未解決参照を解決して実行ファイルを生成する処理を行う。
この処理を行うツールがリンカであり、LLVMではLLDが使用される。

ライブラリには静的ライブラリと動的ライブラリがある。
静的ライブラリ（.a / .lib）はリンク前のオブジェクトファイルをアーカイブにしたものであり、リンカが必要なオブジェクトファイルを取り出して実行ファイルに埋め込む。
アーカイブとは、複数のオブジェクトファイルを1つにまとめたものである。
動的ライブラリ（.so / .dll）はリンク済みの機械語を含む独立したファイルであり、実行ファイルとは別に配布される。
リンク時にはリンカが参照情報のみを実行ファイルに記録し、実行時にOSのローダーがライブラリをメモリに読み込んでアドレスを解決する。


### リンク時最適化
また、リンク時最適化（Link Time Optimization, LTO）を行うこともある。
通常、最適化はコンパイル単位で行われるため、ファイルをまたいだ最適化は難しい。
LTOでは、リンク時にプログラム全体を見渡すことで、コンパイル単位を超えた最適化を行うことが可能になる。
LTOはコンパイラのオプションによって有効にできる。
さらに、オブジェクトファイルにLLVM IRを保持するオプションもあり、これを使えばリンク時にIRレベルの最適化も行うことができる。
ただし、この場合オブジェクトファイルがLLVM IR依存になるため、リンカもLLVM系（LLD）を使う必要があり、GNU系のリンカ（ld）など外部ツールとの互換性が失われる可能性がある。


### ビルドシステム
ファイルが増えると、コンパイルコマンドを手動で管理するのは困難になる。
どのファイルをどの順序でコンパイルするか、どのライブラリをリンクするか、変更されたファイルだけを再コンパイルする差分ビルドなど、管理すべきことが多い。
これを自動化するのがビルドシステムである。

CMakeはビルド定義ツールであり、何をどうビルドするかを`CMakeLists.txt`に記述する。
CMake自体がコンパイルやリンクを行うわけではなく、ビルドルールを生成する役割を担う。

Ninjaはビルドランナーであり、CMakeが生成したビルドルール（`build.ninja`）に従って、実際にClangやリンカを呼び出してビルドを実行する。

```
CMakeLists.txt（ビルド定義）
    │ cmake -G Ninja
    ▼
build.ninja（ビルドルール）
    │ ninja
    ▼
Clang / LLD を呼び出して → 実行ファイル
```
