# C++開発環境
この記事では、Clang/LLVM・CMake・Ninja を中心としたC++開発環境を整理する。


## 登場人物
| ツール | カテゴリ | 役割 |
|--------|----------|------|
| C++ | プログラミング言語 | ソースコードを記述するための言語 |
| Clang | コンパイラ / ドライバ | 前処理・構文解析・意味解析・IR生成を行い、必要に応じてリンカも呼び出す |
| LLVM | コンパイラ基盤 | IRの最適化とコード生成を担う |
| LLD | リンカ | オブジェクトファイルやライブラリを結合して実行ファイルを生成する |
| CMake | ビルド定義ツール | 何をどのようにビルドするかを記述する |
| Ninja | ビルド実行ツール | CMake が生成したビルドルールを高速に実行する |
| Clang-Format | コード整形ツール | コードスタイルを自動でそろえる |
| Clang-Tidy | 静的解析 / リンタ | コード品質やモダンC++の観点から問題点を指摘する |
| CppCheck | 静的解析 | Clang とは独立した解析器で不具合の兆候を検出する |
| GoogleTest | テストフレームワーク | 単体テストを記述・実行する |
| clangd | エディタ支援 | 補完、定義ジャンプ、診断などを提供する |
| Sanitizers | 動的解析 | 実行時にメモリ破壊や未定義動作、データ競合などを検出する |

## コードを書く
C++のソースコードは、主に `.cpp` や `.hpp` などのファイルに記述する。  

> [!NOTE]
> `.cpp` の代わりに `.cc` や `.cxx` が使われることもある。  
> `.hpp` の代わりに `.h`、`.hh`、`.hxx` が使われることもある。

一般的には、ヘッダファイルには宣言やインターフェースを、ソースファイルには実装を置く。  
ただし、テンプレート、`inline` 関数、`constexpr` 関数、クラス定義の一部などはヘッダ側に置かれることも多い。

この段階では、次のような補助ツールがよく使われる。

Clang-Format はコード整形ツールであり、プロジェクト全体のスタイルをそろえるために使う。  
`.clang-format` にルールを定義しておくことで、インデントや改行位置、括弧の配置などを自動で統一できる。

Clang-Tidy は静的解析とリンティングのためのツールであり、コード品質を向上させるために使う。  
モダン C++ のベストプラクティス違反、危険な書き方、冗長な記述などを検出できる。  
`.clang-tidy` に設定を書いておくことで、プロジェクトごとの解析方針を共有しやすくなる。

CppCheck も静的解析ツールだが、Clang-Tidy とは異なる解析器を持つ。  
そのため、Clang-Tidy では見つからない不具合の兆候を補完的に検出できることがある。  
複数の静的解析ツールを併用することで、検出できる問題の幅を広げられる。

clangd は C++ 向けの Language Server であり、エディタ上での補完、定義ジャンプ、参照検索、リネーム、診断表示などを支える。  
単なる補完機能ではなく、「そのプロジェクトが実際にどうコンパイルされるか」を理解したうえで支援を行うのが特徴である。  
このため、`compile_commands.json` を参照してインクルードパス、マクロ定義、コンパイルオプションを把握することが多い。  
エディタ上で早い段階からエラーや警告の候補を見つけられるため、コンパイル前のフィードバックを大きく改善できる。

これらのうち、Clang-Tidy や CppCheck は CI に組み込んで継続的に実行することも一般的である。  
一方で clangd は主に開発者の編集体験を支えるツールとして使われる。

## コンパイルする
C++のコンパイルは、単に「ソースコードを機械語に変える」だけではない。  
大まかには、前処理、解析とIR生成、最適化、コード生成という段階を経て進む。

### 前処理
まず、プリプロセッサが前処理を行う。

前処理では、主に次のような処理が行われる。

- `#include` によるヘッダの展開
- `#define` によるマクロ展開
- `#if`、`#ifdef` などによる条件コンパイル

この結果、コンパイラ本体が解析する単位ができる。  
この単位は翻訳単位と呼ばれる。

### 解析とIR生成
前処理後のコードに対して、Clang が字句解析、構文解析、意味解析を行う。

#### 字句解析
字句解析では、ソースコードをトークンに分割する。  
たとえば `int x = 42;` は、`int`、`x`、`=`、`42`、`;` のようなトークン列として扱われる。

#### 構文解析
構文解析では、トークン列が文法規則に従っているかを確認し、構文木を組み立てる。  
たとえば `int x = 42` のようにセミコロンが欠けていれば、ここで構文エラーになる。

#### 意味解析
意味解析では、型の整合性、名前解決、スコープ、オーバーロード解決などを確認する。  
たとえば `int x = "hello";` は文法上は読めても、型が合わないためエラーになる。

#### IR生成
解析が終わると、Clang はコードを中間表現である IR に変換する。  
Clang/LLVM 系では、代表的な IR として LLVM IR が使われる。

たとえば、次のような関数があるとする。

```cpp
int answer() {
    return 42;
}
```

概念的には、LLVM IR では次のように表現される。

```llvm
define i32 @answer() {
entry:
  ret i32 42
}
```

さらに、x86-64 向けのアセンブリでは概念的に次のようになる。

```assembly
answer:
    mov eax, 42
    ret
```

実際の出力は、最適化レベル、ABI、ターゲットCPU、アセンブラ記法などによって異なる。

### 最適化
生成された IR は、LLVM によって最適化される。  
ここでは、ソースコードの意味を保ったまま、より効率的なコードに変換するための処理が行われる。  
この段階は一般にミドルエンドと呼ばれる。

代表的な最適化には、次のようなものがある。

- 定数畳み込み
- 不要コード削除
- 関数インライン展開
- ループ最適化
- ベクトル化

### コード生成
最適化後の IR から、ターゲット環境向けの機械語を生成する。  
この段階はバックエンドと呼ばれる。

生成結果は通常、オブジェクトファイルとして出力される。  
オブジェクトファイルには機械語だけでなく、シンボル情報や再配置情報なども含まれる。

主な形式は次の通りである。

- Windows: COFF / `.obj`
- Linux: ELF / `.o`
- macOS: Mach-O / `.o`

## リンクしてビルドする
ビルドとは、コンパイルだけでなく、オブジェクトファイルやライブラリを結合して最終成果物を作る工程全体を指す。  
その中心にあるのがリンキングである。

### リンキング
リンカは、生成済みのオブジェクトファイルとライブラリを結合し、未解決参照を解決して実行ファイルやライブラリを作る。

Clang はコンパイラフロントエンドであると同時にドライバでもあり、必要に応じて適切なリンカを呼び出す。  
環境によっては `lld` を使うこともあれば、プラットフォーム標準のリンカを使うこともある。

ライブラリには大きく分けて 2 種類ある。

- 静的ライブラリ: `.a` や `.lib`。必要なオブジェクトコードを実行ファイルに取り込む
- 動的ライブラリ: `.so`、`.dylib`、`.dll`。実行時に OS のローダーが読み込む

なお、Windows では `.dll` に加えて、リンク時に使うインポートライブラリ `.lib` が併用されることが多い。

### リンク時最適化
通常の最適化は、各翻訳単位ごとに行われる。  
そのため、ファイルをまたぐ最適化には限界がある。

そこで使われるのが LTO（Link Time Optimization）である。  
LTO を有効にすると、リンク時にプログラム全体を見渡して追加の最適化を行える。  
代表的な方式として、Full LTO や ThinLTO がある。

Clang/LLVM 系では、オブジェクトファイルに LLVM bitcode を保持してリンク時に最適化する構成も利用できる。  
ただし、その場合はリンカや関連ツールチェーン側にも対応が必要であり、環境によって使える構成や互換性が異なる。

### ビルドシステム
ソースファイルや依存ライブラリが増えると、コンパイルコマンドを手作業で管理するのは難しくなる。  
どのファイルをどのオプションでコンパイルするか、どの順序でリンクするか、変更があったものだけを再ビルドするか、といった管理が必要になる。

これを支えるのがビルドシステムである。

CMake はビルド定義ツールであり、`CMakeLists.txt` にターゲット、ソースファイル、依存関係、コンパイルオプションなどを記述する。  
CMake 自体がコンパイルを行うのではなく、Ninja や Visual Studio プロジェクトなど、各種ビルドシステム向けの定義を生成する。

Ninja は、その生成結果に従って実際のビルドを高速に実行するツールである。

```text
CMakeLists.txt
    │ cmake -G Ninja
    ▼
build.ninja
    │ ninja
    ▼
Clang / リンカが実行される
    ▼
実行ファイルやライブラリが生成される
```

## テストする
C++では、GoogleTest のようなテストフレームワークを使って単体テストを書くことが多い。  
テストコードは通常 `.cpp` ファイルとして記述し、`TEST` などのマクロを使ってテストケースを定義する。

テストを実行するには、テスト対象コードとテストコードを一緒にコンパイルし、テスト用の実行ファイルを作る必要がある。  
多くの場合、本番用の実行ファイルとは別に、テスト専用のターゲットを用意する。

その理由は次の通りである。

- テストフレームワーク側の `main` を使うことがある
- 本番用とテスト用でリンク対象が異なることがある
- 警告設定、サニタイザ、カバレッジ計測などのオプションを変えたいことがある

ここでいうサニタイザとは、AddressSanitizer、UndefinedBehaviorSanitizer、ThreadSanitizer などの実行時検査機構である。  
これらはソースコードを静的に読むのではなく、サニタイザ用の計測コードを埋め込んだ実行ファイルを動かすことで問題を検出する。  
たとえば、配列外アクセス、use-after-free、未定義動作、データ競合などは、通常の単体テストだけでは見逃されても、サニタイザ付き実行で発見できることがある。  
そのため、C++ では「テストを書く」ことと「サニタイザ有効でテストを回す」ことをセットで考える場面が多い。

つまり、テストは「ソースコードを書いて終わり」ではなく、通常のビルド設定の一部として管理される。

## 開発を支える補助ツール
実際のC++開発では、コンパイルとビルド以外にも次のようなツールがよく使われる。

- デバッガ: `lldb`、`gdb`、Visual Studio Debugger など
- Sanitizers: AddressSanitizer、UndefinedBehaviorSanitizer、ThreadSanitizer など
- 依存管理: `vcpkg`、`Conan` など
- CI: GitHub Actions、GitLab CI、Jenkins など

Sanitizers は動的解析ツール群であり、プログラムの実行中にしか見えない問題を検出する。  
静的解析ツールが「ソースコードを読んで怪しい箇所を指摘する」のに対して、Sanitizers は「実際に動かしながら壊れ方を観測する」点が大きく異なる。  
Clang/LLVM 系との相性も良く、開発用ビルドやテスト用ビルドで有効化する運用が広く行われている。

これらを組み合わせることで、C++開発環境は単なる「コンパイラ」ではなく、編集、解析、ビルド、テスト、デバッグ、自動化までを含む一連の仕組みとして成り立っている。

## まとめ
C++開発環境では、ソースコードを書いた後、前処理、解析、IR生成、最適化、コード生成、リンキングという段階を経て実行ファイルが作られる。  
Clang は主にフロントエンドとドライバを担い、LLVM は最適化とコード生成を担う。  
さらに、CMake と Ninja がビルド全体を管理し、GoogleTest や各種解析ツールが品質を支える。

このように、C++の開発環境は複数のツールが役割分担しながら連携することで成り立っている。
